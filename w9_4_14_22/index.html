What Are Databases?

A database is a way to store data so that you can retrieve it or perform functions based on it.

Databases In Hiding

You've most certainly interacted with databases whether you realize it or not. Almost every website, app, and company has a database that it relies on to store information. If you've signed up for a rewards account you're in sombebody's database. If you've created a profile on a website all of that information is stored in their database.

Why Use Databases?

Now that you're aware of the things that are databases all around you, you can probably imagine the many use cases for storing information. Much like APIs can expand what you're capable of building, a database can take things up a notch in the complexity of what you can build. Most importantly, you can store information beyond the user closing the browser/app.

Types of Databases

There are many types of databases, similar to programming languages or frameworks there are always new tools popping up. However, for the most part all of these projects will be a take on one of 2 general database paradigms.

SQL VS NoSQL

These two types of database are Relational (SQL) and Non-Relational (NoSQL) databases.



What is SQL?

SQL is an acronym that stands for Structrued Query Language. This is the language that is used to communicate with relational databases. It is pronounced SQL or "sequel" conversationally. There are a few popular types of SQL servers such as MySQL, MSSQL, and PostgreSQL.

Relational Databases

Information is stored in tables. Tables are made up of rows and columns. We can use specific keys to connect information across tables. This database is great for applications that have a pretty rigidly defined data structure that will be consistent across all items. This means that, once defined, the columns for the tables won't be changed frequently. Relational databases are ACID compliant and so these are the types of databases used by banks and companies that need their information to remain consistent across tables.

ACID

  • Atomic

    All operations in a transaction must succeed or all of the operations will be reverted.

  • Consistent

    On the completion of a transation, the database is structurally sound

  • Isolated

    Means that transactions are not competing with one another, and instead run sequentially.

  • Durable

    The result of transactions are permanent even when a failure occurs

TL;DR

Some applications require data to be consistent and up to date, such as banks or companies that have to keep transaction logs. This can be achieved by using a relational database because of its ACID compliance

Non-Relational Databases

In contrast to relational databases, non-relational data is less structured. Instead of being stored in tables it is stored in collections. These collections contain documents and these documents can be of any structure and are not beholden to a specific table pattern. These documents CAN contain all of the same data but NoSQL allows you the flexibility to only store what is necessary for a given piece of data. Most often used when working with large data sets. Especially when working with complex/diverse data.
